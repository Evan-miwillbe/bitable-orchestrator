# 安全协议详细参考

## 快照机制

### 快照时机

| 操作类型 | 快照策略 |
|---------|---------|
| UPDATE单条 | 快照该记录所有字段 |
| UPDATE批量 | 快照所有受影响记录（分页存储） |
| DELETE单条 | 快照完整记录（含所有字段） |
| DELETE批量 | 快照所有待删除记录 |
| CREATE | 无需快照（回滚=删除） |
| 跨表级联 | 快照所有涉及表的受影响记录 |

### 快照存储格式

```json
{
  "snapshot_id": "snap_20260505_143022",
  "timestamp": "2026-05-05T14:30:22+08:00",
  "operation_desc": "库存扣减：15条订单对应的12条库存记录",
  "tables": {
    "tbl_inventory": {
      "records": [
        {
          "record_id": "rec_xxx",
          "fields": {
            "数量": 100,
            "最后更新": "2026-05-04"
          }
        }
      ]
    }
  },
  "ttl": "7d"
}
```

### 快照生命周期

- 创建：执行前自动创建
- 保留：默认7天
- 清理：过期后自动删除（或用户手动清理）
- 存储位置：`workspace/snapshots/` 目录

## 回滚协议

### 回滚触发条件

1. **执行错误** — 任何步骤返回不可重试的错误
2. **验证失败** — VALIDATE步骤断言不通过（strict模式）
3. **用户中断** — 执行过程中用户主动取消
4. **超时** — 单步操作超过30秒 / 总执行超过10分钟

### 回滚顺序

```
已执行操作: [op1, op2, op3, op4(失败)]

回滚顺序: op3的逆操作 → op2的逆操作 → op1的逆操作

逆操作定义:
- UPDATE的逆操作 = 从快照恢复原始值
- CREATE的逆操作 = DELETE该记录
- DELETE的逆操作 = CREATE（从快照恢复完整记录）
- NOTIFY的逆操作 = 无（不可回滚，发送更正通知）
```

### 部分回滚策略

当LOOP中部分记录成功、部分失败时：

| 策略 | 行为 | 适用场景 |
|------|------|---------|
| strict | 全部回滚 | 数据一致性要求高（财务） |
| tolerant | 保留成功，报告失败 | 批量操作容忍部分失败 |
| best_effort | 跳过失败，继续执行 | 数据清洗/标记类操作 |

默认使用 `tolerant` 策略。用户可在确认阶段覆盖。

## 权限检查清单

### 执行前验证

```
FOR each table in affected_tables:
  ✓ 读取权限（QUERY操作需要）
  ✓ 编辑权限（UPDATE操作需要）
  ✓ 新建权限（CREATE操作需要）
  ✓ 删除权限（DELETE操作需要）
  ✓ 管理权限（仅字段结构变更需要）

IF any permission missing:
  REPORT: "缺少以下权限: {list}"
  SUGGEST: "请联系表管理员授权，或使用更低权限的替代方案"
  ABORT
```

### 最小权限原则

- 只读操作（QUERY/AGGREGATE）→ 仅需读取权限
- 只更新操作 → 读取+编辑权限
- 包含创建 → 读取+编辑+新建权限
- 包含删除 → 全部权限 + 二次确认

## API频率管理

### 限流策略

```
rate_limiter:
  max_requests_per_minute: 100
  max_requests_per_second: 5
  burst_allowance: 10
  
  on_rate_limit:
    strategy: exponential_backoff
    initial_wait: 1000ms
    max_wait: 30000ms
    max_retries: 5
```

### 批量操作优化

| 记录数 | 策略 | 预计耗时 |
|--------|------|---------|
| 1-10 | 逐条执行 | <5秒 |
| 11-50 | batch API（每批10条） | 5-15秒 |
| 51-500 | batch API（每批50条）+ 1秒间隔 | 15-60秒 |
| 501-5000 | batch API（每批50条）+ 2秒间隔 + 进度报告 | 1-5分钟 |
| >5000 | 拒绝/需用户确认分阶段执行 | — |

## 敏感数据保护

### 识别规则

以下字段名模式视为敏感数据：

```
patterns = [
  /密码|password|pwd/i,
  /token|secret|key/i,
  /身份证|id_card|ssn/i,
  /银行卡|card_number|account/i,
  /手机号|phone|mobile/i,  // 日志中脱敏
  /地址|address/i,         // 日志中脱敏
]
```

### 处理策略

- **执行时**：正常处理（需要完整值才能操作）
- **日志中**：脱敏显示（`138****8888`、`6222***1234`）
- **快照中**：完整保存（回滚需要）
- **屏幕输出**：脱敏显示
- **传输中**：不出飞书平台（CLI本地执行）

## 冲突预防

### 并发写入保护

```
执行写操作前:
  1. 读取record.version（如果API支持）
  2. 执行UPDATE时带version条件
  3. 如果version不匹配 → 说明被他人修改 → 重新读取后重试

如果不支持版本号:
  1. 在UPDATE前再次QUERY确认字段值
  2. 如果关键字段已变化 → 报告冲突 → 让用户决策
```

### 飞书原生自动化冲突

当多维表已配置原生自动化时，Orchestrator的操作可能触发二次变更：

```
检测方法:
  - 执行后等待2秒，重新读取记录
  - 如果字段值与写入值不同 → 原生自动化已触发
  
处理:
  - 告警用户："检测到表'{name}'有原生自动化规则，可能与编排操作冲突"
  - 建议：关闭冲突的原生规则，或调整编排逻辑避开
```

## 灾难恢复

### 场景：执行中途断网/crash

```
恢复流程:
  1. 读取 workspace/execution_state.json
  2. 确认最后成功的步骤
  3. 检查快照完整性
  4. 选项:
     a) 从断点继续执行
     b) 回滚已执行部分
     c) 放弃（保留当前状态）
```

### 场景：回滚本身失败

```
极端情况：回滚操作也报错（如权限被撤销）

处理:
  1. 记录回滚失败的具体步骤和原因
  2. 输出手动恢复指南：
     "以下记录需要手动恢复:
      表: {table}, 记录: {id}, 应恢复为: {snapshot_values}"
  3. 保留快照文件（永不自动删除失败回滚的快照）
  4. NOTIFY(admin, "⚠️ 自动回滚失败，需人工介入")
```
