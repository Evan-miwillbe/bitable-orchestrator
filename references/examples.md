# 扩展使用示例

## 示例1：电商订单全流程自动化

```
场景描述：
电商团队用飞书多维表管理订单。当客服确认订单后，需要：
1. 检查库存是否充足
2. 库存充足→扣减库存+更新订单状态为"待发货"
3. 库存不足→标记订单为"缺货等待"+创建补货申请+通知采购
4. 每天18:00汇总今日订单情况发到运营群

用户输入：
"订单表中状态变为'已确认'的订单，检查库存表对应SKU的可用数量。
 库存够就扣减并把订单改为'待发货'。
 不够就标记'缺货等待'，在采购需求表创建记录，通知采购群。
 另外每天下午6点把今天的订单汇总发到运营群。"
```

**编排器生成的执行计划**：

```
Phase A: 即时执行（订单确认时触发）
├── Step 1 [QUERY] 订单表: 状态="已确认" AND 库存处理="未处理"
├── Step 2 [LOOP] 对每条订单:
│   ├── 2.1 [QUERY] 库存表: SKU = 订单.商品编号
│   ├── 2.2 [BRANCH]
│   │   ├── IF 库存.可用数量 >= 订单.数量:
│   │   │   ├── [UPDATE] 库存表: 可用数量 -= 订单.数量
│   │   │   └── [UPDATE] 订单表: 状态="待发货", 库存处理="已扣减"
│   │   └── ELSE:
│   │       ├── [UPDATE] 订单表: 状态="缺货等待", 库存处理="待补货"
│   │       ├── [CREATE] 采购需求表: {SKU, 缺口数量, 关联订单ID, 优先级="高"}
│   │       └── [NOTIFY] 采购群: "⚠️ {SKU}库存不足，缺口{N}件，已创建补货申请"
│   └── 2.3 [UPDATE] 订单表: 处理时间=now()

Phase B: 定时执行（每天18:00）
├── Step 1 [QUERY] 订单表: 创建日期=today
├── Step 2 [AGGREGATE]
│   ├── 总订单数, 已发货数, 缺货数, 总金额
│   └── 按商品类别分组统计
└── Step 3 [NOTIFY] 运营群: 结构化日报消息
```

## 示例2：HR入职流程自动化

```
场景描述：
HR在"候选人表"中把状态改为"已接受offer"时，需要自动触发一系列入职准备：
1. 在"员工信息表"创建记录
2. 在"IT资产申请表"创建设备申请
3. 在"培训计划表"创建入职培训安排
4. 通知相关部门负责人
5. 3天后检查各项准备进度，未完成的自动催办

用户输入：
"候选人接受offer后，在员工表创建记录（把候选人信息带过去），
 申请电脑和门禁卡（写到IT资产表），安排第一周培训（写到培训计划表），
 通知部门leader和IT。3天后检查哪些没完成，没完成的再催一次。"
```

**编排器生成的执行计划**：

```
Phase A: 即时执行
├── Step 1 [QUERY] 候选人表: 状态="已接受offer" AND 入职流程="未启动"
├── Step 2 [LOOP] 对每位候选人:
│   ├── 2.1 [CREATE] 员工信息表: {姓名, 部门, 职位, 入职日, 来源="候选人表"}
│   ├── 2.2 [CREATE] IT资产申请表: {
│   │     申请人=候选人.姓名, 
│   │     设备类型="笔记本电脑+门禁卡",
│   │     需求日期=入职日-3天,
│   │     状态="待处理"
│   │   }
│   ├── 2.3 [CREATE] 培训计划表: {
│   │     学员=候选人.姓名,
│   │     培训周期=入职日~入职日+5天,
│   │     课程=default_onboarding_courses
│   │   }
│   ├── 2.4 [NOTIFY] 部门leader: "新同事{name}将于{date}入职，请准备..."
│   ├── 2.5 [NOTIFY] IT群: "请为{name}准备设备，需求日期{date}"
│   └── 2.6 [UPDATE] 候选人表: 入职流程="已启动"

Phase B: 延迟执行（入职日-3天）
├── Step 1 [QUERY] IT资产申请表: 关联候选人 AND 状态≠"已完成"
├── Step 2 [LOOP] 未完成项:
│   └── [NOTIFY] IT群: "⚠️ 催办：{name}入职在即，设备尚未准备"
└── Step 3 [QUERY] 培训计划表: 关联候选人 AND 讲师="未确认"
    └── [NOTIFY] 培训负责人: "请确认{name}的入职培训讲师安排"
```

## 示例3：项目管理——里程碑联动

```
场景描述：
项目管理表有"里程碑表"和"任务表"。当里程碑完成时：
1. 检查该里程碑下所有任务是否都已完成
2. 全部完成→里程碑标记完成+通知PM
3. 有未完成→列出未完成任务+通知责任人+里程碑标为"逾期"
4. 自动计算项目整体进度（已完成里程碑/总里程碑）写入项目概览

用户输入：
"里程碑到期日到了之后，检查它下面的任务是否都完成了。
 都完成就标记里程碑完成，没完成的列出来催相关人。
 然后更新项目概览里的整体进度百分比。"
```

## 示例4：客户生命周期管理

```
场景描述：
客户表需要根据互动频率自动调整客户等级和跟进策略：
- 30天无互动→降级为"低活跃"+提醒客户经理
- 90天无互动→降级为"流失风险"+通知销售主管
- 新签约7天内→标记为"重点关注"+每天检查首次使用情况
- 续约前30天→自动创建续约提醒任务

用户输入：
"每天早上检查客户表：
 30天没联系的标为低活跃通知客户经理，
 90天的标流失风险通知主管，
 新签7天内每天关注，
 续约前一个月建任务提醒。"
```

**编排器生成的执行计划**：

```
定时执行: 每天 09:00

├── Rule 1 [QUERY] 最后联系日期 < today-30 AND 客户等级≠"低活跃"
│   └── [LOOP] {UPDATE(等级="低活跃"), NOTIFY(客户经理)}
│
├── Rule 2 [QUERY] 最后联系日期 < today-90 AND 客户等级≠"流失风险"
│   └── [LOOP] {UPDATE(等级="流失风险"), NOTIFY(销售主管)}
│
├── Rule 3 [QUERY] 签约日期 > today-7 AND 首次使用="否"
│   └── [LOOP] {NOTIFY(客户经理, "新客户{name}尚未首次使用，请关注")}
│
├── Rule 4 [QUERY] 合同到期日 BETWEEN today+25 AND today+35 AND 续约任务="未创建"
│   └── [LOOP] {
│       CREATE(任务表, {标题="续约跟进-{客户名}", 负责人=客户经理, 截止日=到期日-7天}),
│       UPDATE(客户表, 续约任务="已创建")
│     }
│
└── [AGGREGATE] 今日处理: 降级{N}人, 流失预警{M}人, 续约提醒{K}条
    └── [NOTIFY] 销售群: 今日客户健康概览
```

## 示例5：财务对账自动化

```
用户输入：
"每月5号，把银行流水表和我们的收支记录表做对账：
 金额和日期都匹配的标为'已核销'，
 我们有但银行没有的标为'待确认'，
 银行有但我们没有的创建一条待核实记录。
 对账结果汇总写到月度对账报告。"
```

**编排器处理要点**：
- 双向匹配算法（非简单JOIN）
- 金额容差（±0.01元）
- 日期容差（±1天，考虑跨日结算）
- 一对多匹配（一笔银行流水可能对应多条收支）
- 对账差异必须人工确认才能标记处理

## 复合场景：供应链全链路

```
用户输入：
"我们有4张表：采购订单、入库记录、库存、销售订单。帮我串起来：
 1. 采购到货时，自动在入库记录创建记录，库存+对应数量
 2. 销售订单确认时，库存-对应数量
 3. 库存低于安全线时，自动创建采购订单
 4. 每周五生成库存周报：本周进出、当前库存、低库存预警
 
 所有操作要有审计记录，金额相关的操作需要我确认。"
```

**这个场景展示了**：
- 4张表的循环依赖关系
- 双向操作（加/减库存）
- 自动触发链（库存低→自动采购）
- 防止无限循环（采购→入库→库存增加→不再触发采购）
- 金额操作的人工确认机制
- 定时报表 + 事件触发的混合编排
